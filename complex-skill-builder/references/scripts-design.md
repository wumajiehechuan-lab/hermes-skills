# 脚本设计指南

> 给 Phase 4 使用。Agent 决定是否加 scripts/ 时参考，不一次性加载其他 reference。

---

## 核心判断规则

**涉及精确计算 → 写成脚本。涉及语义判断 → 留在 SKILL.md 规则里。**

| 什么逻辑 | 放哪 | 为什么 |
|---|---|---|
| 数值计算（比例、统计、阈值校验） | `scripts/` | Agent 算数不可靠，脚本一次跑对 |
| 格式校验（JSON schema、文件完整性） | `scripts/` | 确定性规则，脚本比 Agent 人工检查快且准 |
| API 调用封装（固定的请求/响应格式） | `scripts/` | 避免 Agent 手动构造 curl，参数易出错 |
| 环境探测（读 env、检测工具是否存在） | `scripts/` | 确定性操作，Agent 不该猜环境 |
| 文件转换（md→pdf、csv→json 等） | `scripts/` | 固定管道，不需要语义理解 |
| 风格是否匹配 | SKILL.md 规则 | 需要语义判断，脚本做不到 |
| 描述是否充分 | SKILL.md 规则 | 需要理解内容质量，脚本做不到 |
| 模板选择（用哪个模板） | SKILL.md 规则 | 需要匹配用户意图，脚本做不到 |
| 工作流编排（先做 A 还是 B） | SKILL.md 规则 | 依赖上下文判断，脚本做不到 |

## 实际案例

### ✅ 该写成脚本的

**产品出镜率校验**（来自 tvc-director）：
```
需求：统计分镜脚本中产品出现在画面中的比例，低于阈值时报警。
```
→ 写成 `validate_coverage.py`。
原因：这是**数值计算**（占比 = 产品出现次数 / 总分镜数），Agent 计数 20 个分镜容易漏或重复。

**环境模式探测**（来自 gpt-image-2）：
```
需求：检测 OPENAI_API_KEY 是否存在 + ENABLE_GARDEN_IMAGEGEN 开关状态。
```
→ 写成 `check_mode.py`。
原因：这是**环境探测**，Agent 不该猜自己有没有 API key。

**API 调用**（来自 gpt-image-2）：
```
需求：POST /images/generations，参数固定，响应格式固定。
```
→ 写成 `generate.py`。
原因：curl 参数多容易出错，脚本封装后 Agent 一行调用。

### ❌ 不该写成脚本的

**模板选择**：
```
需求：用户说"做个直播带货图"，从 80 个模板中选正确的那个。
```
→ 留在 SKILL.md 的「模板索引 + 匹配逻辑」里。
原因：这是**语义匹配**——「直播带货」→ `live-commerce-ui.md`，脚本无法理解这种映射。

**质量检查**：
```
需求：生成的图片是否符合「真实感」「信息丰富但不杂乱」的要求。
```
→ 留在 SKILL.md 的「反模式防护」里。
原因：这是**语义判断**，脚本只能检查像素尺寸，不能判断观感。

**提问顺序**：
```
需求：用户信息不足时，先问主播身份还是先问商品？
```
→ 留在模板文件的「缺失信息优先提问顺序」里。
原因：这是**上下文决策**，依赖 Agent 对当前对话的理解。

## 脚本文件规范

每个脚本必须有 docstring 说明：
```python
"""
<脚本名> — <一句话用途>

输入: <参数说明>
输出: <返回值/副作用>
使用: python scripts/xxx.py --arg1 value1

触发条件: <Agent 在什么情况下调用此脚本>
"""
```

执行方式：Agent 通过 `terminal()` 调用，不在 SKILL.md 工作流里嵌入脚本源码。

---

## 不该放 scripts/ 的东西

- ❌ 模板内容（那是 references/ 的活）
- ❌ 重复造轮子（已有工具能做的事不写脚本——比如用 `wc -c` 而不是写个 Python 脚本算文件大小）
- ❌ 工作流编排逻辑（Agent 自己会判断先做哪步）
