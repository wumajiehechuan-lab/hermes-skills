# 控制塔设计指南

> 给 Phase 2 使用。Agent 写 SKILL.md 时逐节参考，不一次性加载其他 reference。

---

## SKILL.md 各节怎么写

按此顺序写入，每节约 500-1500 字符：

### 1. Overview
一句话说清 skill 做什么。**不超过 3 句**。如果技能有多模式，这里提一句「支持 X/Y/Z 三种运行模式」。

### 2. When to Use
- 3-5 个触发器（用户说了什么才加载这个 skill）
- 1-3 个排除条件（什么时候**不该**用）
- 格式：bulleted list，不要散文

### 3. 运行模式（如有环境差异）

**先写探测方法**（脚本路径或自检步骤），**再画决策表**：

```markdown
## 运行模式

先运行 `python scripts/check_mode.py` 确定模式。

| 条件 | 模式 | 行为 |
|---|---|---|
| 条件A | Mode X | 完整流程 |
| 条件B | Mode Y | 委托宿主 |
| 条件C | Mode Z | 纯顾问 |

### Mode X
...

### Mode Y
...
```

探测放最前面——Agent 必须先知道自己处于什么环境再往下读。

### 4. Skill 结构

用目录树展示，每项跟一行注释：

```
skill-name/
├── SKILL.md              ← 本文件
├── references/
│   ├── category-a/       ← 用于 XX 场景
│   └── category-b/       ← 用于 YY 场景
└── scripts/
    └── generate.py       ← 核心执行脚本
```

### 5. 模板索引

**只列文件名 + 一句话描述 + 关键词标签，不列内容！**

```markdown
## 模板索引

### 分类 A（`references/category-a/`）
- `template-1.md` — 电商直播带货截图样机（主播+聊天+礼物+商品卡）
  标签：直播, 电商, 带货, 主播, 抖音, 小红书, 淘宝
- `template-2.md` — 社交平台动态详情页（Twitter/X/微博/Threads）
  标签：社交, 截图, 推文, 动态, 评论区

### 分类 B（`references/category-b/`）
- `template-3.md` — 产品爆炸视图海报
  标签：产品, 海报, 爆炸图, 品牌
```

每条索引 = 文件名 + 描述 + 关键词标签。标签让 Agent 做关键词匹配（比读描述文本快得多）。Agent 需要时按路径加载具体文件。

### 6. 工作流

分步流程，标注 [硬检查点]：

```markdown
## 工作流

Step 1: 确定模式 → 探测环境
  ▼
Step 2: 匹配模板 → 从索引定位 → 加载对应 reference
  ▼
**[检查点]** ← 必须停！用户确认参数
  ▼
Step 3: 执行
  ▼
Step 4: 验证 → 输出
```

硬检查点放在不可逆操作前（生成大量文件、调用付费 API 等）。

### 7. 规则与约束

- 输出路径约定
- 文件命名规则（含时间戳模板）
- 何时该问用户、何时该自己做决定

### 8. Common Pitfalls

≥3 条，每条 = 错误行为 + 为什么错 + 正确做法：

```markdown
## Common Pitfalls

1. **错误行为**。为什么错。正确做法：XXX。

2. **错误行为**。为什么错。正确做法：XXX。
```

### 9. Verification Checklist

≥5 项，Agent 完成任务后逐项自检：

```markdown
## Verification Checklist
- [ ] 已跑环境探测
- [ ] 产物已落盘到正确路径
- [ ] 已告知用户输出位置
```

## 决策表写法

**多维度条件 → 用表格，不写散文。**

### 错误（散文）
> 如果用户给了 PRD 且目标明确就直接开始，如果给了 PRD 但目标模糊就问受众，如果没 PRD 但有参考图就提取 tokens……

### 正确（表格）

```markdown
| 场景 | 行为 |
|---|---|
| PRD + 目标明确 | ❌ 不问，直接开始 |
| PRD + 目标模糊 | ⚠️ 只问 1-2 个问题 |
| 只有一句话 | ✅ 深入询问 |
| 有参考图 | ❌ 先提取信息，不够再问 |
```

---

## 硬检查点设计

**在哪放检查点？**
- 不可逆操作前（生成 100 张图之前先确认 1 张样图）
- 信息分叉点（用户的选择会改变后续所有输出）
- 每次最多确认 5 件事（超过拆成两个检查点）

**怎么标注？**
```markdown
**[检查点]** ← 必须停！用 `clarify()` 确认 XXX
```

## 反模式防护

三类：

**禁止短语** — 放在 Common Pitfalls 里：
```markdown
- ❌ 禁止在未跑探测脚本前假设运行环境
- ❌ 禁止把模板索引中的详细内容展开到回复中
```

**强制检查清单** — 放在关键步骤里：
```markdown
### 执行前检查
- [ ] 已加载对应 reference 模板
- [ ] 用户参数已映射到模板字段
```

**错误回退** — 放在对应步骤里：
```markdown
### 调用失败时
API 返回 401/网络错误 → 不重试 → 立即报错并询问用户备选方案
```
